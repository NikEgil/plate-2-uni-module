# Сценарии работы прошивки (по коду `src/main.cpp` и `lib/`)

Документ описывает, **что именно делает устройство** в разных конфигурациях сборки и в разных ситуациях во время работы — построен по фактическому коду, без предположений о намерениях.

Прошивка собирается под конкретную комбинацию трёх флагов в `platformio.ini`:

- `BOARD_TYPE` — роль платы: `0` = узел с датчиками, `1` = шлюз/ретранслятор, `2` = отладочная плата;
- `NET` — канал передачи данных наверх: `0` = только LoRa, `1` = только GSM, `2` = LoRa + GSM;
- `BOARD_REV` — ревизия платы (определяет распиновку, `lib/def/defenitions.h:22-101`).

**Текущая активная сборка** (см. `platformio.ini`): `BOARD_REV=3, NET=2, BOARD_TYPE=1` — это шлюз/ретранслятор (раздел 3.4).

Под каждую комбинацию в `main.cpp` целиком определена своя пара `setup()/loop()` — в файле их пять (строки 1032, 1120, 1190, 1260, 1455/1591). Реально компилируется только одна — та, чьи `#if`-условия совпали с флагами сборки.

## 1. Общий жизненный цикл (для узлов с датчиками, `BOARD_TYPE == 0`)

Все варианты для `BOARD_TYPE == 0` следуют одному паттерну ESP32 deep-sleep:

```
Глубокий сон (esp_deep_sleep_start)
   │   пробуждение по таймеру ИЛИ по одной из кнопок (EXT1 wakeup)
   ▼
setup()
   │  initPins(), Serial, readBatteryVoltage(), RsModbus::init()
   │  loadArrayFromFlash(tableSens)  — какие датчики подключены к портам
   │  stack.begin()                  — открыть FlashStack (очередь пакетов во flash)
   │  loadConfigFromNVS()            — Pass/Broker/Port из энергонезависимой памяти
   │  wake_but = checkButton()       — определить причину пробуждения
   ▼
switch (wake_but):
   ├─ 0 — обычное пробуждение по таймеру  → "рабочий" сценарий (измерить + отправить)
   ├─ 1 — нажата кнопка BUT1              → диагностика канала связи (сигнал)
   ├─ 2 — нажата кнопка BUT2              → переобнаружение датчиков на портах
   └─ 3 — холодный старт / RESET          → полный цикл: сброс + поиск + диагностика
   │
   ▼
sleep(TIME_TO_SLEEP)   — снова в глубокий сон на 15 минут (`TIME_TO_SLEEP = 60*15`)
```

`checkButton()` (`lib/sys/sys.cpp:163-191`) определяет причину так:
- `ESP_SLEEP_WAKEUP_UNDEFINED` (после подачи питания/RESET) → `3`;
- `ESP_SLEEP_WAKEUP_EXT1` и сработал `BUT1` → `1`, `BUT2` → `2` (с ожиданием отпускания кнопки);
- любая другая причина (обычно — таймер) → `0`.

## 2. Узел с датчиками + GSM/MQTT — `BOARD_TYPE==0, NET==1` (`main.cpp:1032`)

```
wake_but
 ├─ 1 → work()                       (на этой плате кнопка 1 не делает спец-диагностику,

 ├─ 3 → cleanUpStack() → searchSensors ×2 → SIM_check_signal() → sleep(10)

```


## 3. Узел с датчиками + LoRa — `BOARD_TYPE==0, NET==0` (`main.cpp:1120` для rev.3, `main.cpp:1190` — общий)

Обе ветки почти идентичны (вторая не вызывает `cleanUpStack()` в сценарии 3):

```
wake_but
 ├─ 1 → lora_check_signal() → sleep(10)
 │        (включить LoRa, отправить тестовый пакет, измерить RSSI ответа, мигнуть силой сигнала)
 ├─ 2 → searchSensors(1), searchSensors(4) → sleep(10)
 ├─ 3 → сброс tableSens в 0, сохранить, [только rev.3: cleanUpStack()],
 │        lora_check_signal(), searchSensors(1), searchSensors(4),
 │        enable_sens(0), enable_power(0) → sleep(10)
 └─ 0 → dataPrepare() → enable_sens(0) → enable_power(0) → lora_send() → sleep(TIME_TO_SLEEP)
```

`lora_send()` (`main.cpp:862-950`) — сеанс передачи накопленных пакетов:
```
включить LoRa-модуль
пока в стеке есть пакеты И сеанс не превысил 2 минуты:
    взять самый старый пакет из стека (с удалением)
    до 2 попыток:
        отправить пакет
        ждать ACK 5 секунд (пакет с правильным device-ID и маркером 0xFF в 4-м байте)
        если ACK получен — установить время по дате из ACK, считать успехом, перейти к следующему пакету
        иначе — пауза 10 секунд, повторить
    если все попытки неудачны — вернуть пакет в конец стека и прервать сеанс целиком
выключить LoRa-модуль
```

## 4. Сквозной блок: измерение датчиков (используется во всех `BOARD_TYPE==0`-сценариях)

```
searchSensors(port):
    включить питание + порт + RS-канал
    1) попробовать опознать "мультисенсор" специальным Modbus-запросом
    2) если не вышло — широковещательный запрос на адрес 0xFF
       код ответа = тип датчика (0x24 = метеостанция, 0x00 = ничего не найдено, иначе — другой датчик)
    сохранить результат в tableSens[port] и во flash (Preferences)

measure(port):
    включить питание + порт + RS-канал
    выбрать ветку по типу датчика (tableSens[port]):
      ├─ 0  → датчик отключён, данных нет
      ├─ 1  → "мультисенсор": опросить адреса 1..5, упаковать валидные ответы (CRC ok) в 50 байт
      ├─ 36 → метеостанция wh65lp: ждать периодической широковещательной посылки,
      │        проверить её собственным CRC8 (полином 0x31)
      └─ иначе → одиночный Modbus-датчик: один опрос регистров, проверка CRC16
    вернуть {data, length, valid}; вызывающий код обязан сам сделать free(data)

dataPrepare():
    собрать заголовок пакета: ID устройства, заряд батареи, текущее время (если валидно)
    измерить оба активных порта (activeport[] = {1, 4}), дописать данные после заголовка,
       при превышении 198 байт — обрезать с предупреждением
    дополнить нулями до фиксированной длины, посчитать CRC
    положить готовый пакет в FlashStack (очередь на отправку)
```

## 5. Шлюз/ретранслятор LoRa → GSM — `BOARD_TYPE==1, NET==2, BOARD_REV==3` (`main.cpp:1260`) — ТЕКУЩАЯ СБОРКА

Назначение по коду: устройство **слушает LoRa-эфир**, принимает пакеты от удалённых сенсорных узлов, подтверждает их получение (ACK с временем), складывает в собственную очередь (`FlashStack`) и периодически переправляет их на сервер по GSM/HTTP.

### 5.1 При старте — встроенный тест отказоустойчивости очереди (БЛОКИРУЕТ обычную работу)

В `setup()` зашит многошаговый тест поведения `FlashStack` при потере питания (`test2_phase`/`test2_crashed`, `main.cpp:1268-1389`), переживающий перезагрузки благодаря `RTC_DATA_ATTR`:

```
test2_phase == 0  → очистить стек, записать 20 тестовых пакетов, sleep(10) [перезагрузка]
test2_phase == 1, !crashed
                  → прочитать первые 7 пакетов, выставить флаг "сбой", sleep(10) [перезагрузка]
test2_phase == 1, crashed
                  → вывести оставшиеся пакеты целиком, перейти в фазу 2
test2_phase == 2  → бесконечный глубокий сон (тест завершён, дальше код не идёт)
```

**Следствие:** пока этот тест не "закончится" (фаза не дойдёт до 2), нормальная рабочая логика (`work()`, приём LoRa) фактически не выполняется — устройство уходит в сон сразу после теста. После завершения теста плата засыпает навсегда (`test2_phase == 2` ведёт только в бесконечный сон), то есть в этой версии файла рабочий цикл шлюза недостижим без ручного сброса этих переменных.

### 5.2 Нормальный рабочий цикл (как он описан в `loop()`, если бы тест не блокировал его)

```
loop() (выполняется постоянно, без глубокого сна):
   приём LoRa-пакета (неблокирующий, receivePacketNB)
      ├─ пакет валиден (CRC ok, длина > 15) → ✅ это данные сенсора:
      │       LORA_sendOK()         — отправить ACK с текущим временем устройства
      │       сохранить пакет в собственный FlashStack
      └─ пакет короткий / CRC не совпал (длина < 15) → 📣 это "пинг" проверки связи:
              LORA_sendOK() без сохранения в стек

   раз в WORK_INTERVAL = 30 минут (или по нажатию BUT2), если в стеке есть пакеты:
       work():
           sim_activate(true)   — включить модем, подключиться к GPRS, прочитать SMS-команды
           http_send()          — переслать накопленные пакеты на сервер (см. раздел 6.3)
           sim_activate(false)
       lora_activate(true)      — снова включить приём LoRa
```

`LORA_sendOK()` формирует ACK-пакет из ID отправителя (взятого из принятого пакета) + маркер `0xFF` + текущее время устройства (если оно «валидно» — фактически проверка `if (isTime)` сравнивает указатель на функцию с нулём и всегда истинна, см. аудит).

## 6. Сквозные блоки передачи данных

### 6.1 LoRa (`NET == 0` или `2`)

- `lora_activate(on/off)` — включить/выключить модуль E220, поднять/опустить UART;
- `lora_send()` — отправка накопленной очереди с подтверждениями (см. раздел 3);
- `lora_check_signal()` — диагностика: отправить тестовый пакет, дождаться ответа, перевести RSSI в "уровень сигнала" 1–5 и показать миганием светодиода;
- `LORA_sendOK()` — формирование и отправка ACK на принятый пакет (используется шлюзом).

### 6.2 GSM/модем (`NET == 1` или `2`, через TinyGSM/SIM800)

- `sim_activate(true)` — до 2 раундов: включить питание модема, инициализировать (`SimModule::begin/activate`), подключиться к GPRS (`connect`), считать качество сигнала и CCID SIM-карты, синхронизировать время по сети (`getNetTime`/`syncSystemClock`), прочитать входящие SMS (`readSMS`);
- `sim_activate(false)` — отключиться от GPRS, деактивировать и выключить модем;
- `SIM_check_signal()` — диагностика сигнала сотовой сети (аналог `lora_check_signal`, миганием показывает уровень от 1 до 5);
- `SIM_reset()` — принудительный сброс модема к заводским настройкам.

### 6.3 Отправка очереди на сервер по HTTP (`http_send`, `main.cpp:801-833`)

```
restart_http()         — переподключиться к серверу (3 попытки, IP захардкожен "158.160.78.215")
для каждого пакета в очереди:
    извлечь пакет из стека
    adding() — обернуть его в "конверт": CCID SIM-карты + ID устройства + заряд батареи
               + уровень сигнала GSM + RSSI последнего LoRa-обмена + сами данные
    httpSendPacketSafe(...) — POST на "/gateway/packets" с заголовком X-Device-Id "device-001"
        ├─ успех → мигнуть индикатором "ок"
        └─ неудача → вернуть пакет обратно в очередь (не теряется)
```

Внутри `httpSendPacketSafe` (`lib/sim/sim.cpp:726-781`) есть до 2 повторов с переподключением к GPRS и пересозданием TCP-соединения при сбоях — данные не теряются, пока хватает места в `FlashStack`.

`mqtt_send()` (закомментирован, `main.cpp:753-788`) — альтернативный путь отправки тех же пакетов через MQTT; код оставлен «на всякий случай», но не активен (и вызывается из недостижимых/несобираемых веток — см. разделы 2 и 3.5 ниже).

### 6.4 SMS-команды дистанционной настройки (`NET == 1`/`2`)

```
readSMS():
    пока есть непрочитанные SMS:
        processOneSMS(текст):
            распарсить как 4 поля: "ID pass broker port"
            если ID совпадает с ID устройства (IDchar) →
                сохранить новые pass/broker/port в NVS (saveConfigFromNVS)
                и сразу применить (loadConfigFromNVS)
```
SMS читаются и удаляются автоматически после обработки. Проверки номера телефона отправителя нет (см. `docs/ai-audit.md`, раздел "Безопасность").

## 7. Старая ревизия шлюза — `BOARD_REV==1, BOARD_TYPE==1` (`main.cpp:1455`)

Структурно повторяет раздел 5, но:
- весь код приёма LoRa и периодической отправки в `loop()` закомментирован — `loop()` фактически ничего не делает, кроме `delay(10); yield();`;
- `setup()` при `wakebut == 3` запускает `SIM_check_signal()` и включает LoRa;
- `work()` вызывает `mqtt_send()` (main.cpp:1470) — функция, чьё тело полностью закомментировано, то есть эта ветка сборки не скомпилируется как есть.

Похоже на «замороженную» промежуточную версию, оставленную в файле ради истории/референса.

## 8. Отладочная плата — `BOARD_REV==3, BOARD_TYPE==2` (`main.cpp:1591`)

Назначение — настройка LoRa-канала через DIP-переключатели и визуальный мониторинг логов через Wi-Fi:

```
setup():
    включить LoRa, прочитать положение переключателей SW1/SW2 → номер канала (1-4)
    мигнуть светодиодом нужное число раз (визуальное подтверждение канала)
    LoRa::configSet(17, channel) — записать канал и адрес в конфигурацию модуля
    поднять Wi-Fi точку доступа "S2-Debug" (открытая сеть с паролем "12345678")
    запустить веб-сервер:
        "/"     → HTML-страница с автообновляемым логом (опрос /log каждые 400 мс)
        "/log"  → отдаёт накопленный буфер последовательного порта (до 3000 байт, см. DualSerial)

loop():
    если нажата кнопка BUT2 → lora_check_signal() (тестовая отправка + измерение RSSI)
    иначе — ничего, просто пауза 1 секунда
```

`DualSerial` (`main.cpp:33-69`) перехватывает весь вывод `Serial.print*` и параллельно копит его в строку `logBuf`, которую веб-сервер раздаёт по `/log` — таким образом весь обычный отладочный вывод устройства становится доступен по сети без подключения кабелем.

## 9. Сводная схема "кто что отправляет и куда"

```
[Сенсорный узел, BOARD_TYPE==0]
   измеряет датчики (RS-485/Modbus)
        │
        ├─ NET==0: пакует пакет → кладёт в свой FlashStack → LoRa → [Шлюз]
        └─ NET==1: пакует пакет → кладёт в свой FlashStack → модем/MQTT → [сервер]  (mqtt_send не реализован)

[Шлюз, BOARD_TYPE==1, NET==2]
   принимает LoRa-пакеты от узлов → подтверждает (ACK + время) → кладёт в свой FlashStack
        │
        └─ периодически по GSM/HTTP оборачивает каждый пакет (CCID, ID, батарея, сигнал, RSSI)
           и отправляет POST на сервер "158.160.78.215/gateway/packets"

[Отладочная плата, BOARD_TYPE==2]
   не участвует в сборе данных — служит для настройки LoRa-канала и просмотра логов по Wi-Fi
```
