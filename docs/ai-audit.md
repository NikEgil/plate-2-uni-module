# Аудит проекта `plate-2-uni-module`

Дата: 2026-06-08
Тип проекта: прошивка ESP32-S2 (lolin_s2_mini) на PlatformIO/Arduino — узел сбора данных с датчиков (RS-485/Modbus), передачей по LoRa и/или GSM (SIM800, TinyGSM), хранением очереди пакетов во flash (`FlashStack`).

Это аудит "только посмотреть" — изменения в код не вносились.

---

## 1. Общая структура

- `src/main.cpp` — 1678 строк, монолитный файл. Содержит **5 разных пар `setup()/loop()`**, выбираемых препроцессором по комбинации трёх флагов `BOARD_TYPE` × `NET` × `BOARD_REV` (main.cpp:1032, 1120, 1190, 1260, 1455, 1591). Это даёт комбинаторный взрыв веток `#if`, тяжело читать и поддерживать — большая часть тела `setup()` дублируется почти дословно между ветками (инициализация пинов, чтение батареи, вывод частоты CPU, загрузка `tableSens`/`stack`/конфига).
- Две почти идентичные функции `measure()` (main.cpp:275 и main.cpp:360) для `BOARD_REV==3` и `BOARD_TYPE==2` — отличаются буквально парой строк (например, `enable_power(0)` в конце, длина пакета метео-датчика). Кандидат на объединение через параметр/флаг.
- `lib/` разбит логично по доменам (`lora`, `sim`, `rs`, `stack`, `sys`, `def`), но вся условная компиляция там же продублирована теми же тремя макросами.
- В корне репозитория лежит каталог **`backups/`** с 9 файлами старых версий прошивки (`main 2.cpp`, `universal module.cpp`, `test field plate 2.cpp`, `recive.tx`, и т.д., суммарно ~140 КБ). Это дублирует то, что и так есть в git-истории, раздувает репозиторий и создаёт риск, что кто-то по ошибке начнёт редактировать не тот файл.
- В `tests/` — смесь старого кода (`old.cpp`, 30 КБ), Jupyter-ноутбуков, Python-парсера и JSON-конфигов датчиков. Юнит-тестов на PlatformIO test runner нет; назначение каталога неочевидно из названия.
- `.idea/` присутствует в рабочем дереве (не в `.gitignore`, в `git status` — untracked) — стоит добавить в `.gitignore`, если используется JetBrains IDE.

## 2. Подозрения на ошибки в коде

| Где | Проблема |
|---|---|
| `lib/sys/sys.cpp:138-160` `readSwitchState()` | Функция не имеет `return` на пути, когда состояние ещё не стабилизировалось (нет `else { return ...; }` после `if (millis() - lastChange > SWITCH_DEBOUNCE_MS)`). Не-void функция может завершиться без `return` → неопределённое поведение / возврат мусора из регистра. |
| `lib/sys/sys.cpp:193-203` `waitForButtonRelease()` | По названию и комментарию должна **ждать отпускания** кнопки, но возвращается немедленно, если кнопка **нажата** (`== LOW`), и ждёт (через `delay`), если кнопка отпущена. Условие выглядит инвертированным относительно описанного поведения. |
| `lib/lora/lora.cpp:222-225, 369-372` `messageGetOKrssi`/`messageGetOK` | Цикл копирования `for (int l = 0; l < len; l++) { mas[i] = (byte)rc.data.charAt(i); }` использует внешний индекс `i` вместо `l` — копирует один и тот же байт `len` раз вместо посимвольного копирования строки в массив. Похоже на copy-paste ошибку. |
| `lib/lora/lora.cpp:228-230, 375-377` | Сравнение ID собрано через одиночный `&` вместо `&&`/скобок: `mas[0] == (byte)(ID >> 16) & 0xFF & mas[1] == ... & 0xFF`. Из-за приоритета операторов это разбирается как побитовое И булевых результатов сравнения — работает "случайно правильно" только потому, что операнды равны 0/1, но крайне хрупко и нечитаемо. Тот же паттерн (с `&&` между сравнениями, но `& 0xFF` внутри каждого) повторён в `main.cpp:906-908, 960-962` — там семантически безопасно (т.к. `&&` разделяет сравнения), но `& 0xFF` всё равно лишний и сбивает с толку. |
| `src/main.cpp:851` `LORA_sendOK()` | `if (isTime) {` — сравнивается указатель на функцию с нулём (всегда true), вместо вызова `isTime()`. Дата в ACK-пакет, видимо, всегда пытается подставляться, даже если время не валидно. |
| `src/main.cpp:1059, 1129, 1199` | `Serial.printf("Rev: %d, NET: %d/%d\n", BOARD_REV, NET);` — в строке формата 3 спецификатора `%d`, передано только 2 аргумента → неопределённое поведение / мусор в выводе. |
| `lib/sim/sim.h:61, 64` | В заголовке продублированы прототипы `static void clearModemBuffer()` и `static bool readFullHttpResponse(int&)`, хотя они объявлены `static` в `.cpp` (т.е. имеют внутреннее связывание и не должны попадать в публичный namespace-интерфейс). Похоже на копипасту комментариев секций из `.cpp` в `.h`; по факту эти строки ничего не меняют (или могут конфликтовать при другой организации TU), но засоряют публичный API. |
| `src/main.cpp:83, 94, 114` | Глобальная `Preferences prefs;` объявлена, но в `loadConfigFromNVS`/`saveConfigFromNVS` создаются **локальные** `Preferences prefs`, затеняющие глобальную. Похоже, глобальная переменная не используется нигде — мёртвый код. |
| `measure()` (main.cpp:275-357, 360-441), `case 1`/`default`/`case 36` | `res.data = (uint8_t*)malloc(...)` нигде не проверяется на `nullptr` перед `memset`/`memcpy` — при нехватке памяти будет падение по null-pointer. На встраиваемой системе с фрагментацией кучи это не чисто гипотетический случай, особенно при долгой работе TinyGSM/HTTP (много `String`). |

## 3. Безопасность

- **SMS-команда смены конфигурации без проверки отправителя.** `processOneSMS()` (main.cpp:549-587) при совпадении `ID` (просто целое число `24`, строка `"00000024"`) сохраняет в NVS новые `pass`, `broker`, `port` — **без проверки номера телефона отправителя**. Любой, кто узнает/подберёт ID устройства (4-значное число, и формат "ID pass broker port" предсказуем), может SMS-сообщением переключить устройство на свой MQTT/HTTP-брокер, фактически перехватив телеметрию или организовав MITM. Стоит как минимум сверять номер отправителя со списком доверенных, либо подписывать команду секретом.
- **Хардкод учётных данных и адресов в исходниках:**
  - APN/логин/пароль GPRS: `"m2m.mts.ru"`, `"mts"`, `"mts"` (`lib/def/defenitions.cpp:44-46`);
  - IP сервера телеметрии `"158.160.78.215"` зашит в двух местах (`src/main.cpp:793, 817`);
  - WiFi точка доступа отладочного режима: SSID `"S2-Debug"`, пароль `"12345678"` — слабый, предсказуемый пароль из 8 символов (`src/main.cpp:74-75`).
  Это нормально для встраиваемого устройства без секретного хранилища, но при компрометации прошивки (например, через дамп флеша) всё это становится известно атакующему; для продакшна обычно выносят в защищённый раздел/подписанный конфиг.
- **Открытый отладочный веб-сервер.** В режиме `BOARD_TYPE == 2` поднимается `AsyncWebServer` на открытой Wi-Fi точке доступа и отдаёт полный лог последовательного порта (включая, потенциально, пароли/брокер из `loadConfigFromNVS`, см. main.cpp:100-103) без какой-либо аутентификации (`/log` доступен всем в сети 192.168.4.0/24). Если этот режим может оказаться в полевой прошивке — это утечка чувствительных данных.
- Конфигурация (`Pass`, `Broker`, `Port`) хранится в NVS открытым текстом через `Preferences` (`saveConfigFromNVS`, main.cpp:113-121) — стандартно для ESP32, но стоит иметь в виду при анализе угроз (физический доступ к устройству = доступ к секретам).

## 4. "Тестовый"/отладочный код в боевой ветке

- Ветка `BOARD_TYPE == 1 and NET == 2 and BOARD_REV == 3` (`src/main.cpp:1260-1453`) почти целиком занята кодом теста отказоустойчивости `FlashStack` при потере питания: `test2_phase`, `test2_crashed`, `writeTestPacket()`, ручная имитация "POWER FAILURE" через `sleep(10)`. Реальная рабочая логика (`work()`, чтение LoRa-пакетов) фактически недостижима, пока `test2_phase` не дойдёт до `2` и тест не "закончится" (а после этого устройство просто бесконечно уходит в глубокий сон). Похоже на код для одноразового HW-теста, который остался в общей сборке и теперь может случайно попасть в прод-прошивку для этой комбинации флагов.
- Большие закомментированные блоки кода внутри `setup()`/`loop()` (например, main.cpp:1300-1397, 1455-1591, и весь `mqtt_send()` на main.cpp:753-788) — затрудняют чтение и провоцируют ошибки при будущих правках (видно, что в одной из веток `work()` всё ещё **вызывает** закомментированный `mqtt_send()`, main.cpp:1040, 1470 — это не скомпилируется, если эта ветка реально активируется).
- В `lib/sys/sys.cpp:492-541` функции `enable_lora`/`enable_sim`/`activate_sim` для `NET==2 and BOARD_REV==1` представляют собой почти полностью закомментированные тела (оставлен только `Serial.println`) — то есть в этой конфигурации управление питанием LoRa/SIM физически не происходит, при этом код выглядит так, будто должен работать.

## 5. Магические числа и хрупкие конструкции

- Калибровка батареи (`CAL_LOW`/`CAL_HIGH`, `lib/def/defenitions.cpp:6-41`) реализована как длинная цепочка тернарных операторов по `ID` — таблица соответствия "серийный номер устройства → калибровочные точки АЦП" зашита прямо в код и требует пересборки прошивки под каждое устройство. `ID == 24` и `IDchar == "00000024"` (main.cpp/defenitions.cpp) задаются раздельно константами разных типов — рассинхронизация при смене ID не будет поймана компилятором.
- Чувствительные тайминги разбросаны магическими числами по всему коду: `delay(10000)`, `delay(5000)`, `SESSION_TIMEOUT = 120000`, `ROUND_DELAY = 3000` и т.д. — без единого места конфигурации, сложно настраивать поведение под разные сети/датчики.
- `RTC_NOINIT_ATTR flash_state_t fstate` в `FlashStack.cpp:40` — содержит три массива по `TOTAL_SECTORS = 736` элементов (`rec_count`, `free_list`, `busy_list`, каждый `uint16_t`) — это ≈4.4 КБ только под эту структуру в RTC slow memory, у ESP32-S2 которой всего 8 КБ, и она также используется другими `RTC_DATA_ATTR`-переменными в `main.cpp` (`wakeups`, `lastWorkTime`, `lastSleepTime`, `test2_phase`, `test2_crashed`). Стоит явно проверить линковщиком, что всё помещается, а не полагаться на то, что "пока работает".
- Множество отладочных `Serial.printf`/`println` с эмодзи разбросаны по горячим путям — расходуют место в flash под строковые константы и время на UART (актуально, учитывая, что устройство и так почти всё время проводит в `delay()`).

## 6. Управление ресурсами / память

- В `measure()` результат `malloc()` копируется через `memcpy`/`memset` без проверки на `nullptr` (см. таблицу выше).
- `DualSerial` (только `BOARD_TYPE == 2`, main.cpp:33-62) аккумулирует весь вывод в `String logBuf` с обрезкой через `logBuf.remove(0, TRIM_SIZE)`. На `String` это операция со сдвигом буфера, при активном логировании (что нормально для отладочного режима) может вызывать частые реаллокации/копирования и фрагментацию кучи на длительной сессии — для отладочного режима, наверное, приемлемо, но не стоит переносить такой подход в боевую прошивку.
- `g_packet[198]`, `a_packet[250]`, `rxBuffer[250]`, `FlashStack::DATA_LENGTH == 198` — размеры буферов синхронизированы вручную в нескольких местах; нет единой константы/проверки на этапе компиляции (`static_assert`), что может привести к молчаливому несоответствию при будущих изменениях формата пакета.

## 7. Положительные моменты

- `FlashStack` — продуманная реализация кольцевого буфера на сырых разделах flash с CRC на запись, восстановлением состояния после сбоя (`recover_state()`) и явной очисткой секторов сразу после освобождения — выглядит как наиболее зрелая и тщательно написанная часть проекта.
- Аккуратная обработка времени (`isTime`/`setTimeFromHexBytes`/`getPackedTimeBytes`) с валидацией диапазонов значений.
- В сетевом коде (`SimModule::connect`, `httpSendPacketSafe`) есть продуманные циклы повторов, сброс watchdog (`esp_task_wdt_reset`) и аккуратное освобождение ресурсов — видно, что проблемы зависаний на модеме уже были учтены и обработаны.

## Итог

Проект — рабочая, явно "боевая" прошивка с заметными следами быстрой итеративной разработки под конкретные платы/конфигурации: большое количество условной компиляции и дублирования между ветками `setup()`, остатки тестового и закомментированного кода в потенциально активных конфигурациях, несколько мест с похожими на баги конструкциями (приоритет операторов, отсутствующий `return`, инвертированная логика ожидания кнопки). Главные риски с точки зрения безопасности — бесконтрольная смена адреса брокера через SMS и открытый отладочный веб-сервер с логами. Структурно стоило бы разнести комбинации `BOARD_TYPE`/`NET`/`BOARD_REV` по отдельным файлам или хотя бы вынести общий код `setup()` в переиспользуемые функции, а каталог `backups/` убрать из репозитория (история и так в git).
