# Аудит прошивки `plate-2-uni-module`

**Дата:** 2026-06-10
**Модель:** Claude Opus 4.8
**Тип:** прошивка ESP32-S2 (lolin_s2_mini), PlatformIO/Arduino. Узлы сбора данных с RS-485/Modbus датчиков + шлюз LoRa→GSM, очередь пакетов во flash (`FlashStack`).
**Объём:** только код прошивки, которая исполняется на устройствах (`src/main.cpp`, `lib/*`). Каталоги `backups/` и `tests/` **не рассматривались** по запросу.
**Характер:** аудит «только чтение», изменения в код не вносились. Документ дополняет существующий `docs/ai-audit.md`, не дублируя его (старые находки, которые уже исправлены в коде — например `if (isTime)` без скобок в `LORA_sendOK` — отмечены отдельно).

Главный фокус — **стабильность**, потому что прошивку нельзя легко перепрошить в поле. Поэтому приоритет отдан тому, что может привести к зависанию, перезагрузке, порче памяти или порче данных в нормальной эксплуатации, а не только к теоретическим атакам.

## 1. Матрица сборок и согласованность «воркфлоу ↔ код»

В `src/main.cpp` пять пар `setup()/loop()`, выбираемых по `BOARD_TYPE × NET × BOARD_REV`. Реально **собираемых** комбинаций меньше, чем кажется:

| Конфигурация | Роль | Состояние |
|---|---|---|
| `TYPE1 NET2 REV3` | Шлюз LoRa→GSM (**текущая сборка** в `platformio.ini`) | Собирается, рабочая |
| `TYPE0 NET0 REV3` / `TYPE0 NET0` | Узел + LoRa | Собирается |
| `TYPE2 REV3` | Отладочная плата (Wi-Fi лог) | Собирается |
| `TYPE0 NET1` | Узел + GSM/MQTT | **Не линкуется** — `work()` зовёт `mqtt_send()` (main.cpp:993), которого нет |
| `TYPE1 REV1` | Старый шлюз | **Не линкуется** — `work()` зовёт `mqtt_send()` (main.cpp:1445) |

> **H6.** Две конфигурации физически не соберутся. Это не «мёртвый код в рантайме», а ловушка: если кто-то выставит `NET=1` в `platformio.ini` (а это заявленный в комментарии режим: `;тип связи 0лора.1сим.2оба`), сборка упадёт на линковке. Либо удалить эти ветки, либо вернуть/реализовать `mqtt_send()`, либо заменить на `http_send()`.

Структурно остаётся проблема из `ai-audit.md`: тело `setup()` дублируется почти дословно между ветками, что и порождает находки вроде H1/H3 (правка в одной ветке не переносится в другие).

## 2. Аудит по сценариям использования (текущая сборка — шлюз)

### Сценарий 2.1 — Холодный старт шлюза (`wakebut == 3`, `setup()` main.cpp:1242)

Поток: `cleanUpStack()` → `sleep(10)` → (далее код настройки LoRa).

> **H2 — настройка LoRa-канала недостижима.** `sleep(10)` (main.cpp:1263) вызывает `esp_deep_sleep_start()`, который **никогда не возвращает управление**. Весь код после него внутри `if (wakebut==3)` — `lora_activate(true)`, `LoRa::configSet(17, ch)`, чтение DIP-переключателей, `saveMSG(...)` — **мёртвый**. После 10-секундного сна устройство перезагружается уже с `wakebut==0` и в этот блок не заходит.
>
> **Следствие для стабильности:** шлюз **никогда не конфигурирует свой LoRa-модуль из прошивки**. Канал/адрес/мощность E220 остаются такими, какими их кто-то записал извне (отладочной платой `TYPE2`). Если модуль E220 заменят, сбросят к заводским или он потеряет конфигурацию — связь с узлами молча оборвётся, и в прошивке нет пути восстановления. Это самый опасный «тихий» отказ для устройства, которое нельзя перепрошить.
>
> Исправление: перенести `sleep(10)` в конец блока (или убрать вовсе), чтобы `configSet` выполнялся; либо вызывать `configSet` безусловно в начале каждого `setup()`.

Дополнительно: на холодном старте `cleanUpStack()` **стирает всю очередь** (736 секторов). Если устройство случайно перезагрузилось по сбросу/просадке питания (а не по таймеру), вся накопленная, но ещё не отправленная телеметрия теряется. Стоит подумать, действительно ли холодный старт должен чистить очередь.

### Сценарий 2.2 — Рабочий цикл шлюза: приём LoRa (`loop()` main.cpp:1318)

```c
int len = LoRa::receivePacketNB(rxBuffer, sizeof(rxBuffer)); // len ∈ [0,250]
...
printHEX(rxBuffer, (int)rxBuffer[0] + 1);          // C2
bool crcOK = checkCRC(rxBuffer, (int)rxBuffer[0]);  // C2
if (crcOK && len > 15) {
    memcpy(packet, rxBuffer, len);                  // C1: packet = g_packet[198]
    ...
}
```

> **C1 — переполнение буфера из радиоэфира (Critical).** `g_packet` объявлен как `uint8_t g_packet[198]` (main.cpp:81), а `receivePacketNB` может вернуть до `maxSize = sizeof(rxBuffer) = 250` байт. При `len > 198` `memcpy(packet, rxBuffer, len)` пишет за пределы `g_packet` (до 52 байт) в соседние глобальные переменные. Это происходит на **любой вход из эфира**, который прошёл CRC и длиннее 15 байт — то есть управляется внешним радиосигналом. Нужно жёстко ограничить: `len = min(len, (int)sizeof(g_packet));` и отбрасывать пакеты с `len > sizeof(g_packet)`.

> **C2 — доверие к длине из пакета (Critical).** `rxBuffer[0]` — это заявленная отправителем длина, ничем не проверенная. `checkCRC(rxBuffer, rxBuffer[0])` и `printHEX(rxBuffer, rxBuffer[0]+1)` используют её как размер. При `rxBuffer[0] == 0xFF` `checkCRC` читает `rxBuffer[254]`/`rxBuffer[253]`, а `printHEX` — до `rxBuffer[255]`, тогда как буфер всего 250 байт → **OOB-чтение** до 6 байт. Любой может передать в эфир кадр с первым байтом `0xFF`. Нужно проверять `rxBuffer[0] <= len` до любого использования.

Логические риски того же цикла:
- **Нет дедупликации.** Если ACK потерян, узел повторяет пакет; шлюз сохраняет дубликат. Сервер обязан дедуплицировать (по ID+время+seq) — это нужно зафиксировать как требование к серверу.
- **Шлюз глух к эфиру в нескольких окнах:** во время `work()` (HTTP-отправка, секунды-минуты), во время `blink(2,750)` (×несколько = единицы секунд блокирующих `delay`), и во время `sleep(10)` каждые ~2 минуты (см. H3). Узлы это переживают за счёт повторов, но это увеличивает их радиоактивность и расход батареи.

### Сценарий 2.3 — Периодическая отправка на сервер (`work()` → `http_send()` → `adding()`)

> **C3 — переполнение `a_packet` (Critical).** В `adding()` (main.cpp:723):
> ```c
> int lgp = (int)g_packet[0];          // до 255
> memcpy(a_packet + 16, g_packet, lgp); // a_packet[250]; 16+255 = 271 > 250
> ```
> Если в очередь попал пакет с `g_packet[0] > 234`, запись выходит за `a_packet[250]`. В текущем формате датчиков длины малы (≤~111 байт), но: (а) защиты нет, (б) пакет в очередь приходит из эфира (C1/C2) и его `[0]` может быть до 234+ при удачном CRC, (в) `memcpy(..., g_packet, lgp)` при `lgp>198` ещё и читает за пределами `g_packet`. Ограничить `lgp` размером `g_packet` и проверять `16+lgp <= sizeof(a_packet)`.

> **H5 — порт соединения захардкожен.** `Port` грузится из NVS и может меняться по SMS (main.cpp:107, 584), но `httpBegin()` (sim.cpp:712) делает `int port = 80;` и игнорирует его. То есть настраиваемый порт — мёртвая настройка, а TLS-порт (443) использовать нельзя в принципе. Либо пробросить `Port` в `httpBegin`, либо убрать иллюзию настраиваемости.

Прочее в этом пути (положительное и риски):
- `httpSendPacketSafe` при неудаче возвращает пакет в очередь (`stack.write(g_packet)`, main.cpp:774) — данные не теряются, это хорошо.
- Но `readFullHttpResponse` (sim.cpp:641) крутит цикл до 20 секунд с `delay(10)+yield()` **без** `esp_task_wdt_reset()`. Если к этому моменту активен task watchdog (`esp_task_wdt_init(8, true)` выставляется в `connect()`, sim.cpp:203, с `panic=true`) и задача loop в него подписана — возможен watchdog-reset во время медленного HTTP-ответа. См. M-WDT.

### Сценарий 2.4 — Управление питанием SIM/периферии

> **H4 — двойное управление пином `EP` и рассинхрон.** Для `REV3/TYPE1` `EP=2`. Им управляют две независимые функции: `enable_power()` (sys.cpp:326) держит флаг `isPowered` и пишет `EP`, и `activate_sim()` (sys.cpp:475) тоже дёргает `EP` напрямую. В `sim_activate(false)` (main.cpp:612) сначала `enable_power(false)` (ставит `EP=LOW`, флаг `isPowered=false`), затем `activate_sim(false)`, которое заканчивается `digitalWrite(EP, HIGH)`. Итог: пин остаётся `HIGH`, а флаг говорит «выключено». При следующем `enable_power(true)` логика `if (act == isPowered) return` сравнивает с рассинхронизированным флагом. Реальное состояние питания становится неопределённым → либо лишний расход батареи (питание не снято перед сном), либо периферия не включается, когда ожидается. Нужно одно владение пином `EP` и единый источник истины состояния.

### Сценарий 2.5 — Время и ACK (затрагивает ВСЕ узлы)

> **H1 — порча времени узлов из-за «отладочного окна».** В `isTime()` (sys.cpp:671-677):
> ```c
> // 🔹 НОВАЯ ПРОВЕРКА: если час 0 И минуты < 30 → возврат false
> if (ti->tm_hour == 16 && ti->tm_min < 30) { ... return false; }
> ```
> Комментарий говорит «00:00–00:30», а код блокирует **16:00–16:29 каждый день**. В это окно `isTime()` ложно возвращает `false`, и `LORA_sendOK()` (main.cpp:804) отправляет ACK с датой по умолчанию `{0x00,0x01,0x01,0x00,0x00,0x00}` = 2000-01-01. Узел в `lora_send()` (main.cpp:877) берёт дату из ACK и вызывает `setTimeFromHexBytes()`, которая **принимает** год `00→2000` (валидация в sys.cpp:713 проверяет только месяц/день/часы) и **переставляет часы узла в 2000-01-01**. После этого вся телеметрия узла идёт с датой 2000 года до следующей удачной синхронизации.
>
> Это не теоретический баг — он срабатывает ежедневно в фиксированном окне. Минимум: убрать это окно целиком (это явный остаток отладки). Дополнительно: `setTimeFromHexBytes`/`LORA_sendOK` не должны рассылать «время» 2000-01-01 как валидное — стоит не класть дату в ACK, если `!isTime()`, и на стороне узла игнорировать год < 2020.

То же по сути: пока шлюз не синхронизировался с GSM (`getNetTime` внутри `sim_activate`), он раздаёт ACK с 2000-01-01 и портит часы всех узлов. После первой удачной GSM-синхронизации время хранится в RTC и переживает 10-секундные сны — но первое окно после полного обесточивания шлюза опасно для всех узлов в радиусе.

### Сценарий 2.6 — Узел: измерение датчиков и отправка (`TYPE0`)

- **C4 — OOB на `tableSens`.** `byte tableSens[5]` (main.cpp:86), но `saveArrayToFlash`/`loadArrayFromFlash` (sys.cpp:271,280) пишут/читают **8** байт: `putBytes("array", data, 8)` и `getBytes("array", data, 8)`. Чтение пишет 8 байт в 5-байтовый массив → портит 3 байта следующих глобалов (`ccid[10]`). UB, хрупко. Привести длину к `sizeof(tableSens)` (или расширить массив до 8, если так задумано).
- **C5 — сломанная защита от переполнения в `dataPrepare`.** main.cpp:475: `size_t remaining = sizeof(packet) - len;` где `packet` — это `uint8_t*`, и `sizeof(packet) == 4`. Значит `remaining = 4 - len` (огромное unsigned), и условие усечения `if (mRes.length > remaining)` **никогда не истинно** → защита от выхода за `g_packet[198]` не работает. Сейчас спасает только то, что суммарные данные с двух портов малы (~100 байт), но при добавлении датчика с бо́льшим `lenreg` будет переполнение. Заменить на `sizeof(g_packet)` (константу буфера, не указатель). Идеально — `static_assert` на размеры буферов.
- **malloc без проверки** в `measure()` (main.cpp:314,333,351 и аналоги): `res.data = malloc(...)` затем `memset/memcpy` без проверки на `nullptr` → падение при нехватке/фрагментации кучи. Для долгоживущего устройства добавить проверку и аккуратный выход (`res.valid=false`).
- **Расход батареи при недоступном шлюзе.** В `lora_send()` (main.cpp:817) на каждый пакет до 2 попыток, между ними `random(5000, 25001)` мс задержки + 5 c ожидания ACK. Если шлюз недоступен, узел проводит десятки секунд с включённым радио на каждое пробуждение (каждые 15 мин) и быстрее сажает батарею. Стоит добавить экспоненциальный backoff/ограничение числа сеансов подряд.
- `random()` не засеян (`randomSeed` не вызывается) → одинаковая последовательность задержек после каждого включения, что ослабляет расхождение по времени при коллизиях нескольких узлов.

### Сценарий 2.7 — Дистанционная настройка по SMS (`processOneSMS`, main.cpp:561)

- **Нет проверки отправителя.** Команда принимается, если первое поле совпадает с `IDchar` (`"00000025"`). Номер телефона отправителя не проверяется. Кто знает номер SIM и ID устройства, тот SMS-сообщением `00000025 evil.host 80 /path` переключает адрес сервера телеметрии (main.cpp:592 → `saveConfigFromNVS`). Это перехват/подмена телеметрии. Минимум — белый список номеров; лучше — подпись команды общим секретом.
- Формат и парсинг устойчивы (4 поля, обрезка), но `gPort` (main.cpp:584) всё равно не применяется из-за H5.

### Сценарий 2.8 — Отладочная плата (`TYPE2`, main.cpp:1566)

Это тоже физическое устройство, поэтому в зоне аудита:

## 3. Память и устойчивость к сбоям (`FlashStack`)

`FlashStack` — самая зрелая часть проекта: CRC на каждую запись, восстановление состояния `recover_state()` после сбоя RTC, стирание секторов сразу при освобождении, FIFO-вытеснение старых записей при переполнении (write при `total>=MAX_RECORDS`). Это хорошо. Замечания:

- **Бюджет RTC-памяти.** `flash_state_t fstate` в `RTC_NOINIT_ATTR` содержит три массива по `TOTAL_SECTORS=736` (`rec_count`, `free_list`, `busy_list`, по `uint16_t`) — суммарно ≈ **4.4 КБ** RTC slow memory (всего 8 КБ на S2, и её делят другие `RTC_DATA_ATTR`). Сейчас, похоже, влезает, но это «на тоненького». Добавить статическую проверку/контроль на этапе линковки, не полагаться на «пока работает».
- **`RTC_NOINIT` ≠ переживает обесточивание.** Очередь переживает deep-sleep (10-сек циклы шлюза), но при полном пропадании питания состояние в RTC теряется → `begin()` пойдёт в `recover_state()` и пересканирует flash по CRC. Это штатно, но `recover_state` использует `static ... entries[MAX_RECORDS]` ≈ 48 КБ в `.bss` — большой статический расход RAM, держать в уме при добавлении других потребителей памяти.
- `clear()` (FlashStack.cpp:366) стирает 736 секторов в цикле без сброса WDT — на холодном старте шлюза (`cleanUpStack`) это длительная блокирующая операция; убедиться, что не упирается в watchdog на «уставшем» flash.

## 4. Watchdog и блокирующие задержки

- **M-WDT.** `connect()` выставляет `esp_task_wdt_init(8, true)` (panic). Несколько длинных операций (`readFullHttpResponse` до 20 c, `httpSendPacket` ожидание до 20 c — sim.cpp:786) крутятся с `delay/yield`, но **без** `esp_task_wdt_reset()`. Если задача подписана на TWDT — риск перезагрузки прямо во время сетевого обмена (то есть в самый чувствительный момент). Пройтись по всем `while`-ожиданиям > 8 c и добавить сброс WDT, как это уже сделано в `SimModule::connect`.
- Повсеместные блокирующие `delay()` (включая `blink()` по 250–1500 мс, `enable_*` по 200–1000 мс) — в `loop()` шлюза превращаются в «слепые» окна для приёма LoRa. Для шлюза, который должен слушать эфир, желательно увести индикацию/паузы в неблокирующую схему.

## 6. Что сделано хорошо

- `FlashStack`: CRC, восстановление после сбоя, стирание секторов, защита очереди от потери при неудачной отправке (возврат пакета в очередь) — продуманная, надёжная основа.
- Сетевой код (`SimModule::connect`, `httpSendPacketSafe`) — циклы повторов, переподключение GPRS/TCP, сброс WDT в нужных местах, аккуратное освобождение (`end()` удаляет `client`/`modem`/закрывает UART). Видно, что зависания модема уже отрабатывались.
- Валидация времени (`setTimeFromHexBytes`, `getNetworkTime` с восстановлением TZ через `mktime`/`gmtime_r`) — аккуратная работа с временными зонами.
- Прошлый баг `if (isTime)` (сравнение указателя) в `LORA_sendOK` — **исправлен** (теперь `isTime()` с вызовом). `docs/flow.md` §5.5 на этот счёт устарел.

> Примечание: `docs/flow.md` §5.1 описывает встроенный тест отказоустойчивости (`test2_phase`), который в **текущем** `main.cpp` уже удалён из `setup()` шлюза (остался только закомментированный stack-тест на main.cpp:1288-1315). Документацию стоит привести в соответствие.
